blace.hub Model Versioning

In order to enforce compatibility and correctness between the models in our hub, the blace.ai C++ SDK and your applications, model artifacts downloadable from the hub follow a certain naming schema.

Type of artifacts

Model Descriptors

Model descriptors have the base name
<model>_<model-version>_<configuration>_<configuration-version>_<submodule>_<submodule-version>_export_format_<export-format-version> with different file endings.

  • <model> The name of the model, e.g. "segment_anything_v2"
  • <model-version> The version of the model, e.g. "v7". This is changed everytime there is some compatibility breaking change like model API updates or in the case of optimizations.
  • <configuration> The configuration of the model, e.g. "hiera_tiny" or "hiera_base" in case of SAM2.
  • <configuration-version> The version of the configuration, e.g. "v2". This is changed e.g. if weights have been updated.
  • <submodule> and <submodule-version> Some models are split into different submodules, to e.g. allow for caching of intermediate results. In case of SAM2 there is a image_embedding and a mask_predictor submodule so an image only has to be processed once and can be prompted for masks based on the cached embeddings. In case of monolithic model (no submodule) <submodule>_<submodule-version> is set to "ALL".
  • <export-format-version> This will be updated in case of changes to internal data formats. Please read Compatibility to learn how to know if a new version of a model is compatible with some version of the sdk.

The following artifacts exist:

  • **.h** The model metadata and identifier encoded in a header file to be embedded in the application at compile time.
  • **.blacemodel** The model metadata bytes in a file, which can be loaded at runtime with blace::util::registerModel. This allows for dynamic model swapping at runtime.
  • **.json** A human-readable descriptor of the models inputs and outputs.

Model Payload

Model payloads of all types always come as <hash>.bin files and you have to make sure they are present in the correct folder during runtime. Please see the Quickstart tutorial for an example.

Compatibility

On the model card page of every model you will find the sdk version the model was tested against (reference). For a given model and the reference blace.ai sdk version v, you can expect:

  • the model runs with version v
  • the model runs with versions n > v (newer) if the changelog of the versions between v and n do not state compatibility breaking changes or warnings
  • the model runs with version o < v (older) if the changelog of the versions between o and v do not state compatibility breaking changes or warnings
